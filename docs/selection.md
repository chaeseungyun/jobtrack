## 2026-02-23

### 1) 백엔드 코드 리팩토링 필요성 검토
- **선택지**:
  - A: route handler → service 구조 유지
  - B: 3-Layered Architecture (controller → service → repository) 도입
- **최종 결정**: **B안 채택**
- **판단 근거**:
  1. **변경 비용 통제 필요성**: 프로젝트는 현재 serverless 환경(Vercel)에서 실행되나, 향후 서버 분리(Express/Nest) 또는 DB 교체 가능성을 열어두고 있음. 도메인 로직이 HTTP/DB/Framework에 직접 의존할 경우, 변경 비용이 비선형적으로 증가함.
  2. **인프라 의존성 분리**: 기존 service가 Supabase에 직접 접근하여 인프라 결합도가 높음. repository interface를 도입함으로써 도메인 계층을 인프라로부터 분리하여 테스트 용이성 및 교체 가능성 확보.
  3. **재사용성**: service가 HTTP에서 분리됨으로써, 다양한 실행 환경에서 재사용 가능.
  4. **아키텍처 일관성**: controller → service → repository로 의존 방향을 고정하여 응집도를 높이고 레이어 간 결합도를 낮춤.
- **리스크**: 보일러 플레이트 증가, 현재 프로젝트 대비 과설계

### 2) 다크 모드 버튼의 hydration error 처리
- **선택지**:
  - A: 클라이언트에서 버튼을 그리기
  - B: 서버에서 버튼을 그리고 css로 처리하여 JS의 개입을 제거하여 hydration 방지
- **최종 결정**: **B안 채택**
- **판단 근거**:
  - mount를 체크하고 클라이언트에서 버튼을 그리는 것은 next-themes에서도 제안하는 패턴이지만 렌더링 시 버튼이 깜빡이는 단점이 있음.
  - 레이아웃 시프트도 좋지 않은 일임.
  - shadcn 홈페이지를 참고하여 svg + css 기반의 아이콘이라는 영감을 얻었고 css는 next-themes가 class을 주입하면 적용되므로 js가 개입하지 않아서 hydration mismatch는 피하면서 좋은 UI를 만들 수 있음.
- **리스크**: design token, tailwind를 알아야 정확한 이해 가능

## 2026-02-22

### 1) 문서화 및 LLM 협업 전략
- **주제**: 개발 기획을 어떻게 관리해야 LLM이 잘 이해하며 다음 개발에 효율적으로 참고할 수 있을까? 사람은 어떻게 문서를 작성해야 할까?
- **최종 결정**:
  - PRD는 human과 LLM이 모두 이해할 수 있도록 구조화한다.
  - PRD에는 비즈니스 요구사항과 고수준 기술 요구사항을 기록한다.
  - 세부 기술 설계는 별도의 아키텍처 문서로 관리한다.
  - progress.md는 학습 및 실행 기록 문서로 직접 작성하고, LLM은 리뷰 역할만 수행한다.
- **판단 근거**:
  1. PRD는 개발의 목적과 방향을 명확히 정의하는 문서이므로 구조적이고 일관된 형식이 필요하다.
  2. 문서가 파편화되면 맥락 유지가 어렵지만, 목적별 분리는 오히려 이해도를 높인다.
  3. LLM은 제안자 역할로 활용하고, 최종 설계 결정은 사람이 명확히 확정해야 한다.
  4. progress 문서는 스스로 이해했음을 증명하는 학습 로그이므로 직접 작성하는 것이 의미 있다.

- **주제**: 이벤트 리소스 통합 및 저장 원자성 확보
- **최종 결정**: 마감일 통합(Form) + 다중 일정 분리(Card Action) 하이브리드 구조

  1. 가장 중요한 데이터인 `deadline`은 지원서 수정 시 한꺼번에 저장(Atomic)되어야 정합성이 보장됨.
  2. 반면 코딩 테스트나 면접 등은 가변적인 다중 일정임으로, 개별적인 생성/수정/삭제 액션이 필요함.
  3. 이를 위해 마감일은 지원서 PATCH 보디에 포함하고, 그 외 일정은 독립된 `api/events`를 통해 관리하는 구조로 이원화하여 사용성과 확장성을 모두 확보함.

- **최종 결정**: **B안**
