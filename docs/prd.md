> - 기술적 구현 방법보다는 "사용자 가치"에 집중하여 작성


> **문서 관리 규칙**

## 1. 프로젝트 개요 (Project Overview)
- **이름**: JobTrack
- **목표**: "취업 준비생이 지원 현황을 한눈에 파악하고, 중요한 면접/코테 일정을 놓치지 않도록 돕는 자동화된 관리 플랫폼"
- **핵심 가치**: 일정 누락 방지, 지원 파이프라인 시각화, 서류 통합 관리.

### 1.1 문제 정의 (Problem Definition)
- 여러 채용 플랫폼(원티드, 프로그래머스, 자체 공고 등)에 흩어진 지원 정보를 관리하기 어려움.
- 면접 일정을 캘린더에 수동으로 등록하는 과정에서 누락이 발생함.
- 현재 자신의 지원 단계(서류, 코테, 면접 등)가 어느 위치에 있는지 시각적으로 파악하기 힘듦.

### 1.2 비즈니스 목표 (Goals)
- **MVP (V1)**: 핵심 지원서 관리 및 일정 리마인더 기능의 안정적 구현.
- **고도화 (V2)**: 사용자 경험(UX) 개선 및 인프라 확장을 위한 아키텍처 안정화.

## 2. 사용자 시나리오 (User Scenarios)

### 2.1 사용자 페르소나

**구직자** — 여러 채용 사이트를 오가며 지원하는 신입 ~ 경력직 구직자로, 흩어진 지원 정보를 한 곳에서 추적하고 입력 부담을 줄이고 싶어한다.

- **대상**: 신입(대학생/부트캠프 수료생) ~ 경력직 이직자
- **행동 패턴**:
  - 잡코리아, 사람인, 원티드 등 복수 채용 사이트를 병행 사용한다.
  - 한 사이클에 3~4개를 동시 지원한다.
  - 지원할 때 JobTrack에 접속하여 기록한다 (매일 접속은 아님).
  - 취업 성공 시 이탈하고, 이직 시 복귀한다.
- **Pain Point (우선순위순)**:
  1. **추적 문제**: 어디에 지원했는지, 어디에 지원할 건지 파악이 안 된다.
  2. **입력 비용**: 노션 등에 수기 입력이 귀찮아서 기록을 포기하게 된다.
  3. **회고 어려움**: 지원 결과를 돌아보며 전략을 조정하기 어렵다.
  4. **일정 누락**: 면접/코테 일정을 놓칠 수 있다 (빈도는 낮지만 치명적).
- **기능 매핑**:
  | Pain Point | 해결 기능 | 단계 |
  |---|---|---|
  | 추적 | 지원서 CRUD, 칸반 보드 | V1~V2 |
  | 입력 비용 | URL 파싱 자동입력 → 확장 프로그램 → Gmail 감지 | V3~V4 |
  | 회고 | 대시보드 KPI, 분석 페이지 | V2 |
  | 일정 누락 | 일정 등록, D-3/D-1 이메일 알림 | V1 |

### 2.2 핵심 사용자 여정 (User Journey)
1. **가입 및 로그인**: 서비스를 시작하기 위해 이메일로 가입하고 로그인한다.
2. **지원서 등록**: 새로운 채용 공고에 지원하면 기업명, 직무, 현재 상태를 등록한다.
3. **일정 추가**: 해당 지원서와 관련된 면접이나 코딩 테스트 일정을 등록한다.
4. **현황 파악**: 칸반 보드나 대시보드를 통해 전체 지원 현황과 다가오는 일정을 확인한다.
5. **서류 관리**: 지원 시 제출한 PDF 이력서나 포트폴리오를 업로드하여 관리한다.
6. **알림 수신**: 중요한 일정 전날이나 당일에 리마인드 이메일을 받는다.

### 2.3 랜딩 페이지 진입 (Landing Entry)
1. **랜딩 페이지 확인**: 로그인하지 않은 사용자는 서비스의 핵심 가치(파이프라인 시각화, 일정 누락 방지, 서류 통합 관리)를 랜딩 페이지에서 빠르게 이해한다.
2. **가입/로그인 결정**: 랜딩 페이지의 CTA를 통해 로그인/회원가입 화면으로 이동한다.
3. **인증 사용자 하이브리드 진입**: 이미 로그인한 사용자도 루트(`/`) 접속 시 기본적으로 대시보드로 리다이렉트된다. 단, 앱 내 푸터의 '서비스 소개' 링크를 통해 `?landing=true` 파라미터로 랜딩 페이지에 접근할 수 있으며, 이때는 '대시보드로 돌아가기' 버튼이 제공된다.
4. **인증 화면 복귀 경로**: `/auth` 화면에서는 보조 링크를 통해 랜딩(`/`)으로 돌아갈 수 있으며, 로그인 성공 시 원래 접속하려던 페이지(`callbackUrl`)로 자동 이동한다. `callbackUrl`은 경계 함수(`getSafeCallbackUrl`)를 통해 상대 경로만 허용되며, 비정상 값은 대시보드(`/dashboard`)로 안전하게 fallback된다.

## 3. 핵심 기능 명세 (Feature Set)

### 3.1 인증 (Auth)
- 이메일 기반 회원가입 및 로그인.
- 안전한 세션 유지 및 접근 제어.

### 3.2 지원서 및 일정 관리 (Applications & Events)
- **지원서 CRUD**: 기업 정보 및 지원 상태 관리.
- **칸반 보드**: 드래그 앤 드롭을 통한 지원 단계 시각화 (`관심 -> 지원완료 -> 서류합격 -> ...`).
- **일정 리마인더**: 주요 일정 D-3, D-1 자동 이메일 알림.

### 3.3 문서 관리 (Documents)
- 지원서별 관련 서류(PDF) 업로드 및 보관.

### 3.4 UX 및 테마
- 대시보드를 통한 핵심 지표 요약 제공.
- 다크모드 지원을 통한 시각적 편의 제공.

### 3.5 데이터 시각화 및 분석 (Analytics)
- **목적**: 사용자가 합격률 개선과 꾸준한 지원 활동을 데이터 기반으로 판단할 수 있게 한다.
- **북극성 지표**: 합격률(서류→면접→최종). **선행 지표**: 주간 지원 건수(꾸준함).
- **대시보드 (`/dashboard`)**:
  - KPI 카드 5종: 전체 지원, 응답률, 서류합격률, 면접전환율, 최종합격률.
  - 단계별 현황 퍼널 차트 (수평 바).
  - 최근 30일 지원 추이 라인 차트.
  - 출처별 성과 비교 그룹 바 차트.
  - analytics 실패 시 기존 3카드 레이아웃으로 graceful fallback.
- **칸반 보드 (`/board`)**:
  - 칼럼 헤더에 다음 단계 전환율 표시 (→ N%).
  - 14일 이상 정체 중인 지원서에 amber 경고 배지 표시.
- **지원서 상세 (`/applications/[id]`)**:
  - 이벤트 타임라인: 시간순 세로 타임라인, D-day 배지, 상대 시간 표시.
  - 과거 이벤트와 미래 이벤트를 시각적으로 구분.
- **심화 분석 (`/analytics`)**:
  - 기간 선택(7일/30일/90일/전체)에 따른 동적 분석.
  - 대시보드와 동일한 차트를 클라이언트 사이드에서 기간별로 재요청.
- **차트 스택**: Recharts + shadcn CSS 변수 기반 테마 연동.
- **금지 원칙**: 시각화만 있고 행동 유도가 없는 '관상용' 지표 추가 금지. 5개 초과 범주 Pie, 3D 차트, 이중축 남용 금지.
### 3.6 랜딩 페이지 (Landing Page)
- **목적**: 로그인 전 사용자에게 JobTrack의 가치를 전달하고, 가입/로그인 전환을 돕는다.
- **구성**: Hero(핵심 문장 + CTA) → 기능 소개(칸반/알림/문서) → 미리보기(보드 흐름) → 푸터.
- **인증 상태별 CTA 정책**:
  - 비로그인 사용자: `로그인/시작하기/무료로 시작하기`를 `/auth`로 연결한다.
  - 로그인 사용자: 기본적으로 대시보드로 자동 리다이렉트된다. 단, 랜딩 페이지 수동 진입 시 주요 CTA를 `대시보드로 이동`으로 전환하여 제공한다.
- **헤더 접근성/편의**: 랜딩 헤더에서 다크모드 전환 토글을 제공한다.
- **콘텐츠 원칙**: 기술 설명보다는 사용 시나리오와 효익(시간 절약, 일정 누락 방지, 복기 용이)을 중심으로 서술한다.
- **확장성 메시지**: LLM 기반 공고 URL 파싱 기능이 도입되었으며, 향후 추가 자동화 기능을 "예정"으로 명확히 표시하여 과장 없이 비전을 제시한다.

### 3.7 크롬 확장 프로그램 (Chrome Extension)
- **목적**: 채용 공고 페이지에서 컨텍스트 스위칭 없이 지원서를 바로 기록할 수 있게 하여, JobTrack 웹은 개인 대시보드(백오피스) 역할에 집중하게 한다.
- **Manifest**: V3 (Chromium 기반 브라우저 호환 — Chrome, Edge, Brave 등).
- **인증**:
  - 미로그인 상태에서 팝업 열면 "JobTrack 로그인" 버튼을 표시한다.
  - 버튼 클릭 시 JobTrack 웹 로그인 페이지(`/auth`)를 새 탭으로 열고, 로그인 완료 후 확장 프로그램에 인증 토큰(30일 유효)을 전달한다.
  - 토큰은 `chrome.storage.local`에 저장하고, API 호출 시 `Authorization: Bearer` 헤더로 사용한다.
- **공고 저장 플로우**:
  - 로그인 상태에서 브라우저 툴바의 확장 아이콘 클릭 → 팝업에서 "이 공고 저장하기" 버튼 표시.
  - `content_script`가 현재 페이지의 HTML을 추출하여 서버의 HTML 파싱 엔드포인트(`POST /api/applications/parse-html`)로 전송한다 (ScrapingBee 미사용).
  - 파싱 결과(회사명, 포지션, 마감일 등)를 팝업 내 폼에 표시하여 사용자가 확인/수정 후 최종 저장한다.
  - 저장 결과는 팝업 내 토스트 메시지로 표시한다.
- **대상 사이트**: MVP에서는 사람인(`saramin.co.kr`), 잡코리아(`jobkorea.co.kr`)를 지원한다. 이후 범용 지원으로 확장을 검토한다.
- **백엔드 변경**: 기존 URL 파싱 엔드포인트는 유지하고, HTML 직접 수신 파싱 엔드포인트와 Bearer 토큰 인증을 추가한다.
- **범위 외 (MVP)**: 모든 웹사이트 범용 지원, 페이지 내 플로팅 버튼, 브라우저 Notification API, 중복 공고 감지, 사이드패널 UI.
- **상세 요구사항**: `docs/extension/requirements.md` 참조.

### 3.8 인증 페이지 진입 UX (Auth Entry UX)
- 인증 페이지(`/auth`)에는 메인 인증 흐름을 방해하지 않는 보조 링크(`홈으로 돌아가기`)를 제공한다.
- 보호된 페이지 접근 시 인증 페이지로 리다이렉트되며, 로그인 완료 후 원래 목적지로 복귀시키는 `callbackUrl` 기능을 지원한다.
- 해당 링크는 비로그인 사용자의 정보 탐색 맥락을 유지하기 위한 보조 진입점으로 동작한다.

## 4. 비기능 요구사항 (Non-Functional Requirements)
- **보안**: 인증된 사용자만 자신의 데이터에 접근 가능.
- **성능**: 대시보드 및 주요 페이지 로딩 2초 이내.
- **파일 업로드**: 10MB 이하의 PDF 파일만 허용.
- **알림 안정성**: 이메일 발송 실패 시 최대 3회 자동 재시도.
- **확장 프로그램 보안**: 토큰은 `chrome.storage.local`에만 저장, content_script에 노출하지 않음. HTML 전송 시 쿠키/로컬스토리지 등 민감 정보 미포함.
- **확장 프로그램 성능**: 파싱 요청~결과 표시까지 5초 이내.
- **확장 프로그램 호환**: Manifest V3 기반, Chromium 브라우저 호환. Chrome Web Store 등록 기준 충족.

## 6. 성공 지표 (Success Metrics)

### 추적 대상 (측정 인프라 확보 시 기준선 관측 예정)
- **유저당 평균 등록 지원서 수**: 핵심 가치(추적) 활용도의 직접 지표.
- **재방문 주기**: 마지막 접속 후 복귀까지 일수. 구직 사이클 특성상 매일 접속은 아니므로, 지원 시점 기준 복귀 패턴을 관측한다.
- **URL 파싱 사용 비율**: 파싱 등록 / 전체 등록. 자동화(V3)가 입력 비용 문제를 실제로 해소하는지 검증하는 지표.
- **확장 프로그램 등록 비율**: 확장 프로그램 경유 등록 / 전체 등록. 확장 프로그램이 주요 입력 채널이 되는지 검증한다.
- **이메일 알림 발송 성공률**: Resend webhook 상태 기반으로 측정.

### 현재 품질 기준 (V3~V4)
- **핵심 플로우**: 지원서 등록 → 칸반 이동 → 일정 알림 수신까지 끊김 없이 동작.
- **URL 파싱**: 잡코리아, 사람인, 원티드 대상 파싱 성공.
- **이메일 알림**: cron 정상 실행 및 이메일 수신 확인.

> **기준선 설정 원칙**: 초기 유저(5~20명) 유입 후 2~4주 데이터를 관측하여 기준선을 확보한다. 목표 수치는 기준선 확보 후 설정한다.
